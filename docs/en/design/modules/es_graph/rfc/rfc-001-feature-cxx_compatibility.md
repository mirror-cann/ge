# RFC-001: C++ ES API Compatibility Design

## Overview and Goals

### Background

In ES (Eager Style) graph construction, to guarantee C++ ES API backward API compatibility, generation tool needs to master IR prototype evolution of different commercial release versions. History prototype library responsible for accumulating and providing these history prototype data, for `gen_esb` to compare differences and generate overload interfaces during generation phase.

### Goals

1. Establish history prototype library protocol, implement cross-version operator IR prototype information archiving and querying
2. Design C++ ES API overload generation mechanism, ensure backward compatibility
3. Resolve overload interface ambiguity problem, provide clear user calling semantics
4. Establish complete generation, release, consumption flow

## Detailed Design

### History Prototype Library Protocol Design

Detailed design reference: [History Prototype Library Protocol Document](../design/history_op_registry_protocol.md)

Key design points:

- **Positioning and Responsibility**: History prototype library is set of long-term maintained protocols and data products, used for cross-version archiving operator IR prototype information
- **Directory Structure**: Organized by operator package, contains version index, metadata and operator prototype data
- **Data Format**: Uses JSON format, contains minimum field set to ensure backward compatibility
- **Generation and Release**: Generated by `gen_esb --extract-history`, released with Ops run package
- **Consumption Method**: Read through file system interface, serves ES or other scenarios

### C++ API Compatibility Design

Detailed design reference: [C++ API Compatibility Design Document](../design/es_cxx_compatibility_design.md)

Key design points:

#### Module Architecture and Responsibility Division

- **History Prototype Library Module**: Responsible for storing and managing history prototype data (Ops component)
- **gen_esb Tool Module**: Responsible for data generation, reading, business logic and code generation (GE component)

#### Overload Generation and Ambiguity Handling

**Core Problems**:

1. C++ default parameter must be at end of parameter list restriction
2. `EsTensorLike` supporting scalar type implicit conversion causing ambiguity

#### Data Flow and Build Process

- **Commercial Release Build**: Simultaneously generates C++ overload interfaces and history prototype library data
- **Non-commercial Build**: Only generates C++ overload interfaces
- **Engineering-Operator-GE Three-party Collaboration**: Passes information through compilation macros, unified packaging release

## Impact Analysis

### Technical Impact

- **Positive Impact**: Provides stable C++ API compatibility guarantee, improves user experience
- **Risk**: Overload ambiguity handling complexity relatively high, needs strict testing verification

### Architecture Impact

- **Module Dependency**: Adds history prototype library module, but responsibility boundary clear
- **Build Process**: Commercial release build process needs adjustment, adds history data generation step

### Compatibility Impact

- **Backward Compatibility**: Through overload mechanism support ensures existing API calls not affected

### Maintenance Cost

- **Protocol Maintenance**: Needs long-term maintain history prototype library protocol
- **Tool Maintenance**: `gen_esb` tool needs continuous maintain overload generation logic

## Alternative Solutions

### Solution B: Version Namespace

- Put different version APIs into different namespaces (such as `ge::es` and `ge::es::v2`)
- **Advantage**: Completely avoids overload ambiguity
- **Disadvantage**: User calling habit changes, Runnable Dump cost relatively high

Note: Runnable Dump refers to dumping graph object in memory into graph construction code, therefore needs to know specifically what kind of graph construction code to write

### Solution Comparison

| Dimension | Solution A (Recommended) | Solution B |
|------|---------------|-------|
| Ambiguity Risk | Low (through foolproof mechanism) | None |
| User Habit | Stable | Need explicit namespace selection |
| Maintenance Cost | Medium | High |
| Dump Cost | Low | High |

**Recommended Solution A**: Under premise of guaranteeing compatibility, maximally maintain user calling habit stable.
